# 功能开发工作流

Mobazha Unified 项目的 7 阶段系统化功能开发方法。

## 核心原则

- **先问清楚再动手** — 在设计前识别所有模糊点和边界情况
- **先理解后行动** — 先阅读理解现有代码模式
- **简洁优雅** — 优先可读、可维护、架构合理的代码
- **使用 TodoWrite** — 全程跟踪进度

---

## 阶段 1：需求理解

**目标**：明确要构建什么

1. 创建包含 7 个阶段的 todo 列表
2. 如需求不清晰，向用户确认：
   - 要解决什么问题？
   - 功能应该做什么？
   - 有什么约束或要求？
3. 总结理解并与用户确认

---

## 阶段 2：代码探索

**目标**：理解相关的现有代码和模式

1. 并行启动 2-3 个 `code-explorer` agent，各关注不同方面：
   - "查找与 [功能] 类似的功能并追踪其实现"
   - "映射 [功能领域] 的架构和抽象"
   - "分析 [现有相关功能] 的当前实现"
2. 阅读 agent 识别的所有关键文件
3. 向用户展示综合发现摘要

---

## 阶段 3：澄清问题

**目标**：填补信息缺口，解决所有模糊点

**关键：此阶段不可跳过**

1. 回顾代码探索发现和原始需求
2. 识别未明确的方面：
   - 边界情况、错误处理、集成点
   - 范围边界、设计偏好
   - 向后兼容性、性能需求
3. **以清晰有序的列表向用户提出所有问题**
4. **等待用户回答后再继续**

---

## 阶段 4：架构设计

**目标**：设计实现方案

1. 并行启动 2-3 个 `code-architect` agent，关注不同方向：
   - 最小改动（最小变更，最大复用）
   - 整洁架构（可维护性，优雅抽象）
   - 务实平衡（速度 + 质量）
2. 向用户展示：
   - 每个方案的简要总结
   - 权衡比较
   - **你的推荐及理由**
3. **询问用户偏好哪个方案**

---

## 阶段 5：实现

**目标**：构建功能

**未获得用户批准前不得开始**

1. 等待用户明确批准所选方案
2. 阅读前面阶段识别的所有相关文件
3. 按选定架构实现
4. 严格遵循代码规范（见 `.cursor/rules/`）
5. 随进度更新 todo

### 架构检查清单

- [ ] 类型定义在 `@mobazha/core/types/`
- [ ] 转换函数在 `@mobazha/core/utils/transforms/`
- [ ] 数据获取 hooks 在 `@mobazha/core/hooks/`
- [ ] UI 组件只通过 props 接收处理好的数据
- [ ] `apps/` 目录中无业务逻辑

---

## 阶段 6：质量审查

**目标**：确保代码质量

1. 启动 `code-reviewer` agent 审查变更
2. 汇总发现，识别最高严重度问题
3. **向用户展示发现并询问处理方式**（现在修复、稍后修复、或直接继续）
4. 按用户决定处理问题
5. 运行验证：

```bash
pnpm validate:quick
```

---

## 阶段 7：总结

**目标**：记录完成的工作

1. 将所有 todo 标记完成
2. 总结：
   - 构建了什么
   - 做了哪些关键决策
   - 修改/创建了哪些文件
   - 建议的后续步骤
